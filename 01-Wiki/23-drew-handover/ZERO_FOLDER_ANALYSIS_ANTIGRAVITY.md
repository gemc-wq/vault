Analysis of 'Zero' Folder and Infrastructure


This document summarizes the findings regarding the `zero` folder, its internal applications, and the data flow between local systems and external services like Amazon MWS and Google BigQuery.


1. System Overview
The `zero` folder is a repository of internal web applications and management scripts developed for **Ecell Global**. The infrastructure is primarily **PHP-based** and hosted on a **local Philippine (PH) network**, using a distributed MySQL database model (Master-Slave replication).


Core Components
Module | Purpose | Key Files
**REGEN** | Global Report Generator (Sales, Inventory, Active Listings) | `REGEN/lib/report_process.php`
**Amazon_Recon** | Amazon listing reconciliation and tracking | `Amazon_Recon/config/config.php`
**Sherlock** | Task tracking and system-wide logging | `sherlock/main.php`
**Amazon MWS** | Direct API integration with Amazon services | `amazon/getamazoninventoryreport.php`
**Monitoring** | Real-time health checks of the "Zero" database cluster | `zero_monitoring.php`












2. Amazon Listings & Inventory Sources


Current Data Sources
The system utilizes specific MySQL **Views** (prefixed with `uv_`) and tables across multiple local servers:
*   **UK Source Table:** `uv_get_amazon_active_listings_uk_regen` (Server: `192.168.20.168`)
*   **US Source Table:**
*   `uv_get_amazon_active_listings_flus_regen` (Florida / Server: `192.168.20.66`)

*   **Raw Integration Table:** `get_flat_file_open_listings_data` (Stores data fetched via MWS reports).


Report Generation
The **Active Listings Report** is generated by the **REGEN** application.
*   **Job Trigger:** A background cron job runs `REGEN/lib/cron_process_request.php` every few minutes.
*   **Export Logic:** The function `getActiveListing()` in `REGEN/lib/report_process.php` queries the views and exports them to `.xlsx` files using the `PHPExcel` library.






















3. Infrastructure & Connector Architecture


Connector Locations
"Connectors" (API integrations) are **embedded directly in the PHP code**. There is no external middleware or cloud-based connector service identified in this codebase.
*   **Amazon MWS Credentials:** Hardcoded in `amazon/myinclude.php`.
*   **Database Connections:** Managed in `REGEN/config2/dbconnect.php` using local IP addresses (PH Local).


Data Residency
*   **All live transactional data** sits in the local Philippine network (192.168.x.x range).
*   **AWS Status:** There is no evidence in the code of a direct "Datastream" to AWS. If replication to AWS exists, it is likely handled at the database layer (MySQL Replication) rather than the application layer.






























4. BigQuery Synchronization (The 2014 Issue)


Why is data in BigQuery outdated?
The investigation revealed a critical pattern of **hardcoded legacy dates** in several key scripts:
*   **`shipmentTracker_test.php`**: Contains `WHERE a.f_date > '2014-05-23'`.
*   **`replicated_activelistings_process.php`**: Uses legacy eBay site mappings that haven't been modified in years.


Analysis of the Synchronizer
If your BigQuery `zero_dataset` is only showing data from 2014, one of the following is occurring:
1.  **Date Filtering:** The ETL (Extract-Transform-Load) tool used to sync data is calling legacy PHP scripts (like the ones identified) that still use 2014 as the "current" starting point.
2.  **Pipeline Failure:** The modern pipeline (SQL to BQ) is broken, and BigQuery is merely displaying the result of the last successful historical backfill performed in 2014.
3.  **Active Table (Local):** The "Zero" system has moved to new table structures (like the `uv_` views), but the BigQuery sync tool is still pointed at older, static tables that are no longer being updated.


















5. Summary of Findings for Immediate Action
1.  **Check the Database Views**: Verify the definition of the `uv_..._regen` views. If they source from newer tables, the BigQuery sync needs to be re-pointed.
2.  **Update Hardcoded PHP Filters**: Audit all scripts for `2014` date strings. These are likely preventing recent data from being processed for export.
3.  **Review the ETL Tool**: Identify if the team is using a third-party tool (like Fivetran, Stitch, or a custom Python script) to move data from the local `192.168` subnet to BigQuery and verify its SQL queries.
